# Performance Logs

## Overview

This project automatically logs all API request performance metrics to `performance-logs.csv`. This file is tracked in Git to showcase the performance improvements achieved with Redis caching.

## Log File Location

```
cache-project/performance-logs.csv
```

## CSV Format

The log file contains the following columns:

| Column | Description | Example |
|--------|-------------|---------|
| **Timestamp** | When the request was made | 2026-05-10 21:01:12 |
| **Method** | HTTP method | GET, PUT, DELETE |
| **Endpoint** | Full API endpoint | /api/redis-cache/user125 |
| **Key** | Extracted cache key | user125 |
| **Status** | HTTP status code | 200, 404, 500 |
| **Duration(ms)** | Time taken in milliseconds | 13 |
| **Cache Type** | Type of caching used | Redis Cache, MySQL Only, MySQL + Redis |
| **Performance** | Performance category | Excellent, Good, Fair, Slow, Very Slow |

## Performance Categories

| Category | Duration | Description |
|----------|----------|-------------|
| **Excellent** | < 20ms | Lightning fast (Redis cache hit) |
| **Good** | 20-100ms | Fast response |
| **Fair** | 100-500ms | Acceptable response |
| **Slow** | 500-1000ms | Needs optimization |
| **Very Slow** | > 1000ms | Requires immediate attention |

## Cache Types

### Redis Cache
- **Duration:** ~5-20ms
- **Source:** Data served from Redis in-memory cache
- **Indicator:** Very fast response times

### MySQL + Redis
- **Duration:** ~50-200ms
- **Source:** First-time query from MySQL, then cached in Redis
- **Indicator:** Moderate response time on first call

### MySQL Only
- **Duration:** ~50-200ms
- **Source:** Direct MySQL query (no caching)
- **Indicator:** Consistent moderate response times

## Example Log Entries

```csv
Timestamp,Method,Endpoint,Key,Status,Duration(ms),Cache Type,Performance
2026-05-10 21:01:12,PUT,/api/redis-cache/user125,user125,200,1500,MySQL + Redis,Very Slow
2026-05-10 21:01:40,GET,/api/redis-cache/user125,user125,200,13,Redis Cache,Excellent
2026-05-10 21:01:45,GET,/api/redis-cache/user125,user125,200,8,Redis Cache,Excellent
```

**Analysis:**
- First PUT: 1500ms (writing to MySQL + Redis)
- Second GET: 13ms (Redis cache hit) - **115x faster!**
- Third GET: 8ms (Redis cache hit) - **187x faster!**

## How It Works

The `RequestTimingInterceptor` automatically:
1. Captures every API request
2. Measures execution time
3. Determines cache type based on endpoint and duration
4. Categorizes performance
5. Appends to CSV file

## Viewing the Logs

### In GitHub
The CSV file is visible directly on GitHub and can be viewed as a table.

### In Excel/Google Sheets
Open `performance-logs.csv` in any spreadsheet application for analysis.

### Command Line
```bash
# View last 10 entries
tail -10 cache-project/performance-logs.csv

# View all entries
cat cache-project/performance-logs.csv

# Count total requests
wc -l cache-project/performance-logs.csv
```

## Performance Analysis

### Compare Cache vs No Cache

**Without Redis (`/api/cache/*`):**
```
GET /api/cache/user1 - 52ms
GET /api/cache/user1 - 48ms
GET /api/cache/user1 - 51ms
Average: ~50ms (consistent, no improvement)
```

**With Redis (`/api/redis-cache/*`):**
```
GET /api/redis-cache/user1 - 200ms (first call, MySQL + cache)
GET /api/redis-cache/user1 - 13ms (cached)
GET /api/redis-cache/user1 - 8ms (cached)
Average after first call: ~10ms (15x faster!)
```

## Monitoring Tips

1. **Check for Slow Requests**
   - Look for "Very Slow" or "Slow" performance
   - Investigate endpoints with > 500ms duration

2. **Verify Cache Effectiveness**
   - Compare first vs subsequent GET requests
   - Redis cache should show < 20ms after first call

3. **Identify Patterns**
   - Which endpoints are most frequently called?
   - Which keys have the best cache hit rates?

## Cleanup

To clear old logs and start fresh:

```bash
# Backup current logs
cp cache-project/performance-logs.csv cache-project/performance-logs-backup.csv

# Clear logs (keeps header)
echo "Timestamp,Method,Endpoint,Key,Status,Duration(ms),Cache Type,Performance" > cache-project/performance-logs.csv
```

## Integration with CI/CD

You can analyze these logs in your CI/CD pipeline:

```bash
# Check for slow requests
awk -F',' '$6 > 1000 {print}' performance-logs.csv

# Calculate average response time
awk -F',' 'NR>1 {sum+=$6; count++} END {print "Average:", sum/count, "ms"}' performance-logs.csv
```

**Note:** This file is automatically generated and updated by the application. Each request to `/api/cache/*` or `/api/redis-cache/*` endpoints will add a new entry.
